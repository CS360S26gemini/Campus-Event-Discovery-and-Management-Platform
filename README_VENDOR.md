# Vendor Management System Documentation

## 1. Overview
The Vendor Management module is a critical component of the Campus Event Discovery platform, designed to bridge the gap between event organizers and service providers (catering, AV, logistics, etc.). It provides a structured workflow for organizers to request vendor participation and for system administrators to audit and approve these requests.

---

## 2. Architecture & Design Patterns

### 2.1 Pattern: Repository Pattern
All Firestore operations are abstracted through `EventRepository.java`. This ensures:
- **Centralized Logic:** Single point of truth for database interactions.
- **Testability:** Mocking data sources becomes easier.
- **Decoupling:** UI components (Activities/Fragments) don't need to know the structure of Firestore.

### 2.2 Pattern: Parallel Execution (Task-based)
To resolve latency issues, the dashboard uses `Tasks.whenAllComplete()` to fetch Event Proposals and Vendor Requests simultaneously, moving away from sequential callbacks.

## 3. Data Model (`Vendor.java`)

| Field | Type | Description |
| :--- | :--- | :--- |
| `vendorId` | String | Unique UUID generated by Firestore. |
| `vendorName` | String | Commercial name of the vendor. |
| `phoneNumber` | String | Primary contact for the organizer. |
| `vendorType` | String | Category (e.g., Catering, Security). |
| `eventId` | String | Reference to the associated event. |
| `organizerId` | String | UID of the requesting organizer. |
| `status` | String | Enum: `pending`, `approved`, `rejected`. |
| `createdAt` | Timestamp | Server-side generation time. |

---

## 4. Implementation Details

### 4.1 Organizer Workflow
1.  **Request Initiation:** Organizers access the `AddVendorBottomSheet` from the `VendorManagementActivity`.
2.  **Validation:** Client-side checks ensure `vendorName` and `phoneNumber` are populated.
3.  **Submission:** Data is pushed to the `vendor_requests` collection with a `pending` status.

### 4.2 Administrator Workflow
1.  **Dashboard:** The `HomeAdminFragment` displays a "Vendor Requests" section.
2.  **Parallel Loading:** Uses high-performance Task-based queries to reduce load times.
3.  **Moderation:** Admins can Approve/Reject requests. Approval updates the status to `approved`, making the vendor visible to the organizer.

### 4.3 Performance Optimizations
- **Indexing:** Composite indexes on `status` and `createdAt` ensure `O(log n)` query performance.
- **Local Caching:** Enabled via `FirebaseFirestoreSettings.setPersistenceEnabled(true)` to provide instantaneous UI rendering on re-entry.

## 5. UI Components

| Component | Responsibility |
| :--- | :--- |
| `AddVendorBottomSheet` | Handles input and submission of new vendor requests. |
| `VendorManagementActivity` | Displays the list of approved vendors for a specific event. |
| `HomeAdminFragment` | Unified dashboard for admins to manage approvals. |
| `AdminVendorAdapter` | Specialized RecyclerView adapter for programmatic row generation and action handling. |

---

## 6. Security Considerations
- **Role-based Access:** Only users with `role: "admin"` can trigger `approveVendorRequest`.
- **Validation:** All inputs are trimmed and sanitized before being committed to the database.
